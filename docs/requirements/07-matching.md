# マッチング - 要件定義

## 1. モジュール概要

### 1.1 目的
SES案件および受託開発案件に対し、自社社員と協力会社/個人事業主の両方を対象として、スキル・稼働状況・単価レンジを組み合わせたマッチングを行い、営業担当者の候補者選定業務を効率化する。

### 1.2 対象ユーザー
| ロール | 操作権限 |
|--------|----------|
| 管理者 | 全機能（参照・マッチング実行・提案リスト管理） |
| 営業 | 全機能（参照・マッチング実行・提案リスト管理） |
| 事務 | 参照のみ（アサイン確定情報の確認目的） |
| 社員 | 自分がマッチング候補・提案対象になった案件の通知受信のみ |
| 協力会社/個人事業主 | 自分が候補・提案対象になった案件の通知受信のみ |

### 1.3 マッチング対象
- **技術者（供給側）:** 自社社員 / 協力会社技術者 / 個人事業主
- **案件（需要側）:** SES案件 / 受託開発案件（ともにアサイン未確定のもの）

---

## 2. マッチングロジック

### 2.1 マッチングスコアの概念

マッチング候補者は以下3軸のスコアを組み合わせて評価する。スコアは0〜100点で表現し、重み付けは案件種別・営業方針によって調整可能とする。

| 評価軸 | 内容 | デフォルト重み |
|--------|------|----------------|
| スキル適合度 | 案件必要スキルと候補者スキルの一致度 | 50% |
| 稼働状況適合度 | 候補者の稼働可能期間と案件期間の一致度 | 30% |
| 単価適合度 | 候補者の希望単価と案件支払単価レンジの一致度 | 20% |

### 2.2 スキル適合度の算出

- 案件の必須スキルタグと候補者のスキルタグを比較
- 必須スキルの充足率をベースにスコア算出
- 経験年数・スキルレベル（初級 / 中級 / 上級）も加味
- スキルタグの体系はTeam B（社員・スキル）管理のマスタに準拠

**算出例:**
```
必須スキル: [Java(中級以上), Spring Boot, AWS]
候補者スキル: [Java(上級), Spring Boot(中級), Docker]
→ Java: ○（上級は中級以上を満たす）
→ Spring Boot: ○
→ AWS: ✗（未保有）
→ 必須スキル充足率: 2/3 = 66.7%
→ スキル適合スコア: 67点
```

### 2.3 稼働状況適合度の算出

- 候補者の稼働可能開始日と案件の開始予定日を比較
- 候補者の稼働可能終了日（または現在の参画案件の終了日）と案件終了予定日を比較
- 期間が完全に一致または余裕がある場合は高スコア
- 期間の一部しか対応できない場合は低スコア

**稼働ステータス定義（Team B参照）:**
- `available`: 即座に稼働可能
- `available_soon`: 1ヶ月以内に稼働可能
- `in_project`: 現在参画中（終了日次第で調整可能）
- `unavailable`: 稼働不可

### 2.4 単価適合度の算出

- 案件の支払単価レンジ（下限〜上限）と候補者の希望単価（または実績単価）を比較
- レンジ内に収まる場合は高スコア
- レンジを超える場合は超過率に応じてスコアを減点

### 2.5 スコアリングの調整・カスタマイズ

- 営業担当者が案件ごとに各軸の重みを手動調整可能
- 特定スキルを「必須」「あれば尚可」に分類して重み付けを変更可能
- 管理者は重みのデフォルト値をシステム設定で変更可能

## 3. 機能要件

### 3.1 候補者検索

#### 3.1.1 案件起点の候補者検索
- 案件を選択した状態でマッチング検索を実行
- マッチングロジック（2章）に基づき候補者リストを生成
- 候補者ごとにスコア・スキル充足状況・稼働状況・単価情報を表示
- フィルター絞り込み（稼働ステータス・スキル・単価・自社/協力会社の区別）

#### 3.1.2 フリー検索（逆引き）
- スキルタグ・稼働ステータス・単価レンジを指定して候補者を直接検索
- 特定案件に紐づかない候補者プール全体への検索

#### 3.1.3 検索結果の表示
- スコア順 / スキル適合度順 / 単価順でのソート
- 自社社員と協力会社を区別した表示（タブ切り替えまたはフィルター）
- 候補者カードに：氏名・所属（自社/協力会社名）・主要スキル・稼働可能日・希望単価・総合スコアを表示

### 3.2 提案リスト作成

#### 3.2.1 提案リストへの候補追加
- 検索結果から候補者を選択し、提案リストに追加
- 1案件に対して複数の提案リストを作成可能（例：第一候補リスト / 補欠リスト）
- 提案リストに候補追加時のコメント記入（社内メモ）

#### 3.2.2 提案リストの管理
- 提案リストのステータス管理（作成中 / 顧客提案済み / 確定 / 却下）
- 提案リストのPDF出力（顧客提示用：スキルシート抜粋・経歴サマリ）
- 提案リストの複製（類似案件への流用）

#### 3.2.3 候補者への通知
- 提案リストに追加された候補者（社員・協力会社）への通知送信
  - 通知内容：案件概要・期間・就業場所・作業内容（単価は含まない）
- 候補者の意向確認（OK / NG / 保留）の回収機能
- 意向回収のリマインダー設定

### 3.3 マッチング結果管理

#### 3.3.1 マッチング結果の記録
- 各案件のマッチング実施履歴を保持
- 最終的にアサインされた候補者と、落選した候補者・理由を記録

#### 3.3.2 アサイン確定処理
- 提案リスト上の候補者を「アサイン確定」に変更
- 案件管理モジュール（06-project）のアサイン情報に自動反映
- 確定時に単価情報（支払単価）の設定を促す

#### 3.3.3 マッチング精度の改善
- 過去マッチング結果のフィードバック記録（なぜ落選したか）
- フィードバックデータを将来のスコアリング改善に活用（将来拡張）

## 4. データ項目定義

### 4.1 マッチング検索条件（matching_searches）テーブル

| 項目名 | 型 | 必須 | 説明 |
|--------|----|------|------|
| id | UUID | ○ | 主キー |
| project_id | UUID | ○ | 対象案件（projects.id） |
| searched_by | UUID | ○ | 検索実行者（users.id） |
| skill_weight | DECIMAL(3,2) | ○ | スキル適合度の重み（0〜1） |
| availability_weight | DECIMAL(3,2) | ○ | 稼働状況適合度の重み（0〜1） |
| rate_weight | DECIMAL(3,2) | ○ | 単価適合度の重み（0〜1） |
| filter_conditions | JSON | - | 絞り込み条件のスナップショット |
| searched_at | TIMESTAMP | ○ | 検索日時 |

### 4.2 マッチングスコア（matching_scores）テーブル

| 項目名 | 型 | 必須 | 説明 |
|--------|----|------|------|
| id | UUID | ○ | 主キー |
| search_id | UUID | ○ | matching_searches.id |
| candidate_type | ENUM | ○ | employee / partner |
| candidate_id | UUID | ○ | users.id または partners.id |
| total_score | DECIMAL(5,2) | ○ | 総合スコア（0〜100） |
| skill_score | DECIMAL(5,2) | ○ | スキル適合スコア |
| availability_score | DECIMAL(5,2) | ○ | 稼働適合スコア |
| rate_score | DECIMAL(5,2) | ○ | 単価適合スコア |
| skill_match_detail | JSON | - | スキルごとの充足状況詳細 |

### 4.3 提案リスト（proposal_lists）テーブル

| 項目名 | 型 | 必須 | 説明 |
|--------|----|------|------|
| id | UUID | ○ | 主キー |

| result | ENUM | - | assigned / rejected / withdrawn（最終結果） |
| rejection_reason | TEXT | - | 落選理由（フィードバック） |

### 4.5 候補者通知（candidate_notifications）テーブル

| sent_at | TIMESTAMP | ○ | 送信日時 |
| channel | ENUM | ○ | email / in_app |

## 6. 他モジュールとの連携インターフェース

### 6.1 提供するデータ・API

| 連携先 | 提供内容 | インターフェース |
|--------|----------|-----------------|
| 案件管理（Team C / 06-project） | アサイン確定情報（候補者ID・参画期間） | アサイン確定API |
| KPIレポート（Team E） | マッチング件数・確定率・リードタイム | 集計API |

### 6.2 受け取るデータ・依存関係

| 連携元 | 受け取る内容 | 用途 |
|--------|-------------|------|
| 認証・権限（Team A） | ログインユーザー情報・権限 | 操作権限チェック・通知送信制御 |
| 社員・スキル（Team B） | 社員スキル情報・稼働状況・希望単価 | スキル適合スコア・稼働適合スコアの算出 |
| 案件管理（Team C / 06-project） | 案件情報（必要スキル・期間・支払単価レンジ） | マッチング検索条件の読み込み |
| 協力会社/個人事業主情報 | 協力会社技術者のスキル・稼働状況・単価 | 社員と同様のスコアリング |

### 6.3 スキルタグの共通化
- スキルタグの体系はTeam B（社員・スキル）が管理するマスタを参照する
- 案件登録時の必要スキルタグと候補者のスキルタグは同一マスタから選択する
- タグ体系変更時はTeam Bと協議の上、マッチングスコア計算への影響を確認する

## 7. ユースケース

### UC-MTH-01: 案件に対するマッチング検索の実行
**アクター:** 営業  
**事前条件:** 対象案件が登録済み（ステータス：商談中〜受注）、Team BのスキルマスタとSES候補者情報が最新  
**主フロー:**
1. 案件詳細画面から「マッチング検索」を起動
2. スキル・稼働・単価の重みを確認（デフォルト値でそのまま使用可）
3. 必要に応じて絞り込み条件を設定（稼働ステータス: availableのみ等）
4. 検索を実行し、スコア順に候補者リストを表示
5. 各候補者のスキル充足状況・稼働カレンダー・単価を確認
6. 気に入った候補者を提案リストに追加

---

### UC-MTH-02: 提案リストの作成と候補者への通知
**アクター:** 営業  
**事前条件:** UC-MTH-01でマッチング検索を実施済み  
**主フロー:**
1. 提案リスト編集画面で候補者の優先順位を設定
2. リスト名（例：「第一候補」）を設定し保存
3. 「候補者に通知」を選択
4. 通知内容（案件概要・期間・就業場所）を確認（単価は含まれないことを確認）
5. 通知を送信。メールとシステム内通知の両方で送信
6. 意向回収期限を設定し、自動リマインダーを有効化

---

### UC-MTH-03: 候補者の意向回収とアサイン確定
**アクター:** 営業  
**事前条件:** 候補者に通知済み  
**主フロー:**
1. 提案リスト詳細画面で各候補者の回答状況（OK / NG / 保留 / 未回答）を確認
2. 第一候補がOKの場合：その候補者を「アサイン確定」に変更
3. アサイン確定フォームで参画開始日・終了予定日・支払単価を設定
4. 確定後、案件管理モジュールのアサイン情報に自動反映される
5. 他候補者はリスト上で「落選」に変更し、理由を記録

**代替フロー:**
- 2a. 第一候補がNGの場合：次の候補者に通知を送り、同様のフローを繰り返す
- 2b. リスト上の全候補がNGの場合：再マッチング検索を実施

---

### UC-MTH-04: 協力会社/個人事業主のマッチング
**アクター:** 営業  
**事前条件:** 協力会社・個人事業主の情報（スキル・稼働・単価）が登録済み  
**主フロー:**
1. マッチング検索時に「協力会社/個人事業主を含む」オプションを有効化
2. 検索結果に自社社員と協力会社が混在して表示される
3. フィルターで「協力会社のみ」「自社社員のみ」「全員」に切り替え可能
4. 以降はUC-MTH-01〜03と同様のフロー

## 8. 制約・前提条件

### 8.1 マッチング対象の制約
- 稼働ステータスが「unavailable（稼働不可）」の候補者は検索結果に表示しない（設定でオーバーライド可能）
- アサイン確定済みで参画中の候補者は重複アサインを警告表示する（禁止ではなく警告）
- 候補者の個人情報（住所・生年月日等）はマッチング画面に表示しない

### 8.2 単価情報の制約
- 候補者への通知には支払単価・顧客請求単価を含めない
- マッチング画面上の単価表示は管理者・営業・事務のみ（社員・協力会社は単価列を非表示）

### 8.3 スキルデータの依存
- マッチングスコアの精度はTeam B（社員・スキル）が管理するスキルデータの質に依存する
- スキルタグが未登録の候補者はスキル適合スコアが0点となり、検索結果の下位に表示される

### 8.4 性能要件
- マッチング検索の結果表示：3秒以内（候補者1,000名以上を想定）
- 提案リストのPDF生成：10秒以内

### 8.5 前提条件
- Team B（社員・スキル）のスキルマスタとスキルデータAPIが利用可能であること
- Team A（認証・権限）の通知送信権限設定が完了していること
- 協力会社/個人事業主の情報登録（スキル・稼働・単価）の責任者・登録フローを事前に決定すること
- マッチングスコアのアルゴリズムは初期バージョンを確定した上で運用開始し、フィードバックを収集して継続改善する方針とする
