# 通知・ワークフロー - 要件定義

## 1. モジュール概要

### 1.1 目的
通知・ワークフローモジュールは、全業務モジュールに横断する基盤機能として、承認フロー管理・期限アラート・各種通知の一元配信を担う。各業務モジュールはこのモジュールを通じて承認依頼・通知の送受信を行うため、個別モジュールは通知ロジックを持たない。

### 1.2 設計方針
- **横断基盤**: 全業務モジュール（認証・社員・案件・勤怠・請求・契約・書類・ダッシュボード）が本モジュールのAPIを呼び出して通知・承認依頼を発行する
- **チャネル抽象化**: アプリ内通知・メール・将来的なSlack等の外部連携を抽象化し、各モジュールはチャネルを意識せず通知を発行できる
- **非同期処理**: 通知送信は非同期キュー（メッセージキュー）で処理し、業務処理のパフォーマンスに影響しない
- **監査証跡**: すべての通知・承認操作は履歴として保持し、後から追跡可能とする

### 1.3 対象ユーザー

| ロール | 主な利用内容 |
|--------|------------|
| 管理者 | 全ワークフロー管理・通知設定・承認者設定 |
| 営業 | 案件関連通知の受信・承認操作 |
| 事務 | 勤務表・経費・請求の承認操作、期限アラート受信 |
| 社員 | 自身の承認依頼提出・結果通知受信 |
| 協力会社/個人事業主 | 勤務表・請求承認依頼の提出・結果通知受信 |

### 2.1 各ワークフローの承認段階

#### WF-001: 勤務表承認（2段階）
```
社員/協力会社 → [提出] → 事務（1次承認）→ [承認/差戻] → 管理者（最終承認）→ [承認/差戻]
```

#### WF-002: 経費承認（2段階）
```
社員 → [申請] → 事務（1次承認）→ [承認/差戻] → 管理者（最終承認）→ [承認/差戻]
```

#### WF-003: 契約更新承認（1段階）
```
事務 → [申請] → 管理者（最終承認）→ [承認/差戻]
```

#### WF-004: 請求書承認（1段階）
```
事務 → [申請] → 管理者（最終承認）→ [承認/差戻]
```

#### WF-005: 協力会社請求承認（2段階）
```
協力会社 → [提出] → 事務（1次確認）→ [確認/差戻] → 管理者（最終承認）→ [承認/差戻]
```

#### WF-006: 案件受注承認（1段階）
```
営業 → [申請] → 管理者（最終承認）→ [承認/差戻]
```

#### WF-007・WF-008: 書類関連承認（1段階）
```
申請者 → [申請] → 管理者（最終承認）→ [承認/差戻]
```

## 3. 通知チャネル

### 3.1 チャネル一覧

| チャネルID | チャネル名 | 実装時期 | 説明 |
|-----------|----------|---------|------|
| CH-001 | アプリ内通知 | MVP（初期リリース） | システムUI上のベルアイコン・通知一覧 |
| CH-002 | メール | MVP（初期リリース） | 登録メールアドレスへの送信 |
| CH-003 | Slack | 将来拡張（v2.0） | Slack Incoming Webhookによるチャンネル通知 |
| CH-004 | SMS | 将来拡張（検討中） | 緊急アラートのみ対象 |

### 3.2 チャネル別仕様

#### CH-001: アプリ内通知
- ヘッダーのベルアイコンに未読件数バッジを表示する
- 通知一覧は新しい順で表示（最大100件、それ以上は「もっと見る」でページネーション）
- 既読・未読の区別をする
- 通知クリックで関連画面にリダイレクトする

#### CH-002: メール通知
- 送信元アドレス：`no-reply@{会社ドメイン}`
- 件名テンプレートは通知種別ごとに定義する
- HTMLメールとテキストメールの両形式で送信する
- 送信失敗時は最大3回リトライする（30分間隔）

#### CH-003: Slack（将来拡張）
- Incoming Webhook URLをシステム設定で管理する
- チャンネルはワークフロー種別ごとに設定可能とする

## 4. 機能要件

### 4.1 承認ワークフロー管理

#### 4.1.1 承認依頼の作成
- 各業務モジュールは通知モジュールのAPIを呼び出して承認依頼を作成する
- **API**: `POST /workflows/requests`
- 入力パラメータ：ワークフロー種別・申請者ID・申請内容（参照URL）・期限日・承認者設定

#### 4.1.2 承認操作
- 承認者はアプリ内通知またはメールのリンクから承認画面に遷移する
- 承認画面では「承認」「差戻」「保留」の操作が可能
- 差戻時はコメント（理由）の入力を必須とする
- 承認操作後、次の承認者または申請者へ通知を送信する

#### 4.1.3 代理承認
- 承認者が不在（休暇等）の場合、管理者が代理承認者を設定できる
- 代理承認者設定は有効期間を指定する

#### 4.1.4 一括承認
- 事務・管理者は同種ワークフローを一覧から複数選択して一括承認できる
- 一括承認の操作ログは個別に記録する

#### 4.1.5 承認期限超過
- 承認期限の24時間前にリマインド通知を送信する
- 承認期限超過時に管理者へエスカレーション通知を送信する
- エスカレーション後72時間経過しても未対応の場合、管理者に再度通知する

### 4.3 通知設定管理

#### 4.3.1 個人通知設定
- ユーザーは通知種別ごとに受信チャネルを選択できる（アプリ内・メール）
- 通知をオフにできるのは「任意通知」のみとし、業務上必須の通知（承認依頼等）はオフにできない
- メール受信時間帯を設定できる（例：業務時間内のみ）

#### 4.3.2 システム全体通知設定（管理者のみ）
- 各ワークフローの承認者（デフォルト）を設定する
- アラート通知のスケジュール（日時）を設定する
- メールテンプレートを編集する
- 将来的なSlack Webhook URLを設定する

## 8. ユースケース（主要承認フロー別）

### UC-NTF-001: 社員が月次勤務表を提出して承認を受ける（WF-001）

1. 社員が勤怠管理画面（Team D）で月次勤務表を完成させ「提出」ボタンを押す
2. Team D が通知モジュールの承認依頼APIを呼び出し、WF-001の承認リクエストを作成する
3. 事務担当に「勤務表承認依頼」の通知が送信される（アプリ内 + メール）
4. 事務担当が承認タスク一覧（SCR-NTF-002）から該当タスクを選択し、承認詳細（SCR-NTF-003）で承認操作を行う
5. 管理者に2次承認通知が送信される
6. 管理者が承認操作を行う
7. 社員に「勤務表承認完了」通知が送信される
8. Team D の勤務表ステータスが「承認済」に更新される（イベント連携）

### UC-NTF-002: 勤務表が差し戻される

1. UC-NTF-001のステップ4で事務担当が「差戻」を選択し、差戻理由を入力する
2. 社員に「勤務表差戻」通知が送信される（アプリ内 + メール）
3. 通知に差戻理由と勤務表へのリンクが含まれる
4. 社員が勤務表を修正して再提出する（UC-NTF-001に戻る）

### UC-NTF-003: 契約更新期限アラートを受信する（ALT-001・002）

1. 毎朝9:00にスケジュールバッチが実行される
2. 契約終了日の60日前に該当する契約を検索する
3. 対象契約ごとに事務・管理者へアラート通知を送信する（アプリ内 + メール）
4. 事務担当が通知クリックで契約管理画面に遷移し、更新手続きを開始する

### UC-NTF-004: 管理者が承認者不在時に代理承認者を設定する

1. 管理者が代理承認設定画面（SCR-NTF-008）を開く
2. 不在の承認者・代理承認者・有効期間を設定する
3. 不在期間中、該当承認者宛の承認依頼は代理承認者にも通知される
4. 代理承認者が承認操作を行うと、承認操作ログに代理承認者IDが記録される

### UC-NTF-005: 承認期限超過時のエスカレーション

1. 承認期限の24時間前にリマインド通知が承認者に送信される
2. 承認期限を超過すると、管理者にエスカレーション通知が送信される
3. エスカレーション後72時間経過しても未対応の場合、管理者に再通知される
4. 管理者が強制承認または代理承認者を設定して対応する

## 9. 制約・前提条件

### 9.1 パフォーマンス
- アプリ内通知の表示：1秒以内
- 承認操作後の次承認者への通知送信：30秒以内（非同期）
- メール送信：5分以内（送信キューに積まれてから）
- スケジュールバッチ（毎朝9:00）：1時間以内に完了

### 9.2 信頼性
- 通知送信失敗時は最大3回リトライする（30分間隔）
- リトライ失敗時は管理者にエラーアラートを送信する
- ワークフローリクエストのデータは承認完了後7年間保持する（電子帳簿保存法対応）

### 9.3 セキュリティ
- 承認操作は操作者のJWTロールを検証してから実行する
- 他ユーザーの承認タスクは閲覧・操作できない（管理者を除く）
- 通知本文に個人情報を含める場合はマスキングを検討する

### 9.4 前提条件
- メール送信にはSMTPサーバーまたはSES（AWS Simple Email Service）を使用する
- 通知の非同期処理にはメッセージキュー（例：AWS SQS・RabbitMQ等）を使用する
- Slack連携はv2.0以降の実装とし、Incoming Webhook URLの設定のみ初期実装に含める
- 承認依頼を発行する各業務モジュールは本モジュールのAPIキーまたはサービスアカウントを使用して呼び出す
