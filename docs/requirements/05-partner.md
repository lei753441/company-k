# 協力会社・個人事業主管理 - 要件定義

**バージョン**: 1.0  
**作成日**: 2026-04-26  
**担当チーム**: Team B

---

## 1. モジュール概要

### 1.1 目的

IT開発会社（SES・受託開発）が業務委託先として活用する「協力会社」および「個人事業主（フリーランス）」の情報を一元管理する。社内社員（モジュール02）とは別に管理し、法的・管理上の違いを適切に反映した情報管理を実現する。

### 1.2 背景・重要性

SES・受託開発会社では、自社社員だけでなく協力会社社員やフリーランスエンジニアを案件にアサインするケースが多い。これらの外部人材は、社員とは異なる法的関係（雇用契約ではなく業務委託契約）にあるため、管理上の明確な区別が必要である。

### 1.3 スコープ

| 対象 | 内容 |
|------|------|
| 対象ユーザー | 管理者、事務、営業 |
| 管理対象1 | 協力会社（法人）およびその所属エンジニア |
| 管理対象2 | 個人事業主（フリーランスエンジニア） |
| 対象外 | 自社社員（→ モジュール02で管理） |

### 1.4 主要機能サマリ

- 協力会社マスタ（法人情報）の登録・管理
- 協力会社所属エンジニアの登録・管理
- 個人事業主（フリーランス）の登録・管理
- スキル情報の管理（社員と同様の体系でTeam Cに提供）
- 契約条件（単価・支払条件・契約形態）の管理
- 取引評価の記録・管理
- 協力会社/個人事業主向けのポータルアカウント管理（Team A連携）

## 3. 協力会社と個人事業主の違い（管理上の差異）

SES・受託開発の実務において、協力会社（法人）と個人事業主（フリーランス）は法的・管理上の扱いが大きく異なる。本モジュールではこの差異を明確に反映する。

### 3.1 法的・契約上の差異

| 観点 | 協力会社（法人） | 個人事業主 |
|------|----------------|-----------|
| 契約相手 | 法人（会社） | 個人 |
| 契約形態 | 業務委託契約（準委任・請負） | 業務委託契約（準委任・請負） |
| 請求書発行者 | 法人（法人番号あり） | 個人（インボイス番号の有無が重要） |
| 消費税処理 | 原則課税（仕入税額控除の対象） | インボイス登録事業者か否かで異なる |
| 社会保険 | 法人が責任を持つ | 個人が自己管理（国民健康保険等） |
| 偽装請負リスク | 法人管理下のエンジニアへの指揮命令が問題になりうる | 特に注意が必要 |
| 管理単位 | 法人 + 所属エンジニア（1対多） | 個人（1人が1レコード） |

### 3.2 インボイス制度対応（個人事業主固有）

2023年10月以降の適格請求書等保存方式（インボイス制度）への対応として：

- 個人事業主の「適格請求書発行事業者登録番号（T-xxxxxxxxxxxxxxx）」を管理する
- インボイス番号未登録の個人事業主からの請求に対しては、消費税の仕入税額控除が制限されることを担当者が認識できるよう警告表示を行う
- インボイス番号は登録申請中・登録済み・未登録の3ステータスで管理する

### 3.3 管理上の差異サマリ

| 管理項目 | 協力会社（法人） | 個人事業主 |
|---------|----------------|-----------|
| 法人番号 | 必須 | 不要 |
| インボイス番号 | 法人のインボイス番号（法人は原則課税） | 個人のインボイス番号（要確認） |
| 所属エンジニア管理 | 必要（1法人：多エンジニア） | 不要（個人がエンジニア本人） |
| 担当窓口 | 会社の担当者（営業・経理等） | 本人 |
| 名刺・会社情報 | 必要 | 不要（屋号があれば管理） |
| 就業規則・服務規程 | 所属会社に依存 | 本人の裁量 |
| 契約書の締結 | 法人代表者との契約 | 本人との契約 |
| 請求書フォーマット確認 | 法人フォーマット | 個人フォーマット（インボイス要件確認） |

### 3.4 偽装請負リスク管理

| 機能 | 説明 |
|------|------|
| 指揮命令記録 | 協力会社・個人事業主への業務指示の形式（成果物指示か工数指示か）を記録する |
| 警告表示 | 指揮命令関係が疑われる契約形態の場合、警告を表示する |
| 契約形態チェック | 準委任契約と請負契約で管理画面上の表示を切り替える |

## 6. 他モジュールとの連携インターフェース

### 6.1 Team A（認証・権限）との連携

| 方向 | イベント/API | タイミング | 内容 |
|------|-------------|-----------|------|

## 7. ユースケース

### UC-PTN-01: 新規協力会社の登録

**アクター**: 事務担当、営業担当  
**前提条件**: 取引開始の合意が得られ、基本契約書が締結済みであること

**主フロー**:
1. 事務担当が協力会社登録画面（SCR-PTN-003）にアクセスする
2. 会社情報（会社名・法人番号・担当者情報・支払条件）を入力する
3. 「登録」ボタンをクリックし、バリデーションが通過する
4. 協力会社IDが採番され、登録が完了する
5. 所属エンジニアの登録画面（SCR-PTN-005）に遷移し、エンジニア情報を登録する
6. エンジニアのシステムアカウント作成をTeam Aに依頼する
7. エンジニアにログイン情報が通知される

---

### UC-PTN-02: 個人事業主（フリーランス）の登録

**アクター**: 事務担当  
**前提条件**: 個人事業主との業務委託契約書が締結済みであること

**主フロー**:
1. 事務担当が個人事業主登録画面（SCR-PTN-009）にアクセスする
2. 基本情報・連絡先・インボイス登録番号・支払条件・銀行口座を入力する
3. インボイス登録番号が未登録の場合、警告メッセージを表示する（消費税仕入税額控除不可の旨）
4. 「登録」ボタンをクリックして登録を完了する
5. システムアカウント作成をTeam Aに依頼する

**代替フロー**:
- 3a. インボイス番号未入力でも登録は可能（警告表示のみ）。後日登録が判明した時点で更新する

---

### UC-PTN-03: 外部人材のスキル情報更新（自己更新）

**アクター**: 協力会社エンジニア、個人事業主（ポータルから）  
**前提条件**: ポータルにログイン済みであること

**主フロー**:
1. 外部人材がポータル（SCR-PTN-016）にログインする
2. 自己情報編集画面でスキル・資格・プロジェクト経歴を更新する
3. 「保存」ボタンをクリックする
4. スキルシートのバージョンが更新される
5. 担当営業に変更通知メールが送信される（稼働状況変更時のみ）

---

### UC-PTN-04: 取引終了処理

**アクター**: 事務担当  
**前提条件**: 取引終了の意思確認が完了していること

**主フロー（協力会社との取引終了）**:
1. 事務担当が協力会社詳細画面（SCR-PTN-002）から取引停止処理を選択する
2. 取引終了日・終了事由を入力する
3. 所属エンジニアのうち、現在案件アサイン中のエンジニアがいる場合は警告を表示する
4. 確認ダイアログで「確定」をクリックする
5. 協力会社ステータスを「terminated」に変更する
6. 所属エンジニア全員のアカウント無効化をTeam Aに依頼する
7. Team Cに取引終了通知を送信する（アサイン引き継ぎ処理を促す）
8. Team Dに最終請求処理を通知する

---

### UC-PTN-05: 案件終了後の外部人材評価

**アクター**: 担当営業  
**前提条件**: 外部人材のアサイン案件が終了していること

**主フロー**:
1. Team Cからアサイン終了通知を受信し、担当営業に評価登録アラートが送信される
2. 担当営業が評価登録画面（SCR-PTN-013）にアクセスする
3. 技術力・コミュニケーション・信頼性・総合評価（各1〜5点）を入力する
4. コメント・再依頼意向を入力する
5. 「登録」ボタンをクリックして保存する
6. 評価が外部人材の詳細画面に反映される

## 8. 制約・前提条件

### 8.1 性能要件

| 項目 | 要件 |
|------|------|
| 外部人材スキル検索API | 1秒以内（Team Cマッチングエンジンからの呼び出し） |
| 一覧画面表示 | 3秒以内 |
| 同時ポータルアクセス | 100ユーザー以上を想定 |

### 8.2 セキュリティ要件

- 個人事業主の銀行口座情報は暗号化（AES-256）して保存する
- 外部人材の個人情報（住所・銀行口座等）は管理者・事務のみ閲覧可能
- 協力会社エンジニア・個人事業主は自身の情報のみ閲覧・編集可能（ポータル経由）
- 他の外部人材の情報は一切閲覧不可
- 単価情報は管理者・事務のみ閲覧可能（営業は閲覧不可）

### 8.3 法的・コンプライアンス要件

- 個人情報保護法に準拠した個人情報管理を実施する
- インボイス制度に対応した適格請求書発行事業者番号の管理を行う
- 取引終了後の情報保持期間は7年間（商法・法人税法の帳簿保存期間に準拠）
- 偽装請負防止のため、指揮命令関係が生じる業務形態を警告する仕組みを設ける
- 個人事業主との契約が「雇用」と見なされないよう、契約書・管理形態の適切な運用を補助するガイドラインをシステム内に表示する

### 8.4 データ整合性

- 協力会社が「terminated」になった場合、所属エンジニアは自動的に非アクティブにする
- アサイン中の外部人材を取引終了状態にする場合は、必ず担当者への確認と引き継ぎ処理を経ること
- partner_person_idは一度発行したら変更不可
- 単価情報は上書きではなく履歴として積み上げ管理し、適用期間で管理する

### 8.5 前提条件

- Team A（認証）モジュールが先行して稼働していること
- Team B 社員管理モジュール（モジュール02）が構築済みであること（担当営業の紐付けに使用）
- Team C（案件マッチング）のAPI仕様が確定していること
- Team D（勤怠・請求）の支払処理フローが確定していること
- インボイス制度への対応方針が社内で確定していること

### 8.6 運用上の考慮事項

- 協力会社・個人事業主の情報入力は事務担当が中心となり、本人自己入力は限定的な項目のみとする
- 取引先情報の変更は変更履歴をすべて保持し、監査対応を可能にする
- 評価情報は評価した担当者のみ編集可能とし、管理者は閲覧可能とする
