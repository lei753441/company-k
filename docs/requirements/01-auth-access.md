**作成日**: 2026-04-26
**担当チーム**: Team A
**ステータス**: 初版ドラフト

## 1. モジュール概要

### 1.1 目的

本モジュールは、IT開発会社（SES・受託開発）の社内管理システム全体における「誰が」「何に」「どのようにアクセスできるか」を一元的に管理する基盤機能を提供する。社内ユーザー（管理者・営業・事務・社員）と外部ユーザー（協力会社・個人事業主）を明確に区別し、業務上必要最小限のアクセス権を付与するロールベースアクセス制御（RBAC）を実現する。

### 1.2 スコープ

| 対象 | 内容 |
|------|------|
| 含む | ログイン認証、MFA、SSO、ユーザーCRUD、ロール定義、権限設定、セッション管理、監査ログ、APIトークン管理 |
| 含まない | 各業務モジュールの画面・データ自体（Team B〜E が担当）。権限チェックの実行ポイントは各モジュールに委譲するが、権限定義は本モジュールが管理する |

### 1.2A 技術前提（READMEを正とする）
- 認証基盤は **Amazon Cognito** を採用し、ログイン・MFA・SSO・トークン発行は Cognito の機能を優先利用する
- アプリケーション側は Cognito が発行する JWT を検証し、**認可（RBAC/スコープ/自己のみ判定）**と監査を担う
- 永続データストアは **Amazon DynamoDB** を前提とする（権限・監査・APIトークン・補助メタ情報等）

### 1.3 ユーザー区分

```
[内部ユーザー]              [外部ユーザー]
 ├── 管理者                  └── 協力会社担当者
 ├── 営業                        └── 個人事業主
 ├── 事務
 └── 社員
```

外部ユーザーは専用ポータル（サブドメインまたはパス分離）経由でアクセスし、内部業務画面へのアクセスは原則禁止とする。

### 2.1 ロール定義

| ロールID | ロール名 | ユーザー区分 | 説明 |
|----------|----------|-------------|------|

## 3. 機能要件

### 3.1 認証機能

#### 3.1.1 ログイン認証

| ID | 要件 |
|----|------|
| AUTH-001 | メールアドレス＋パスワードによるログイン認証を提供する |
| AUTH-002 | パスワード保管・検証は Cognito に委譲し、アプリケーション側でパスワードハッシュを保持しない |
| AUTH-003 | ログイン失敗5回で当該アカウントを30分間ロックする（管理者は即時解除可能） |
| AUTH-004 | ログイン失敗回数・ロック状態は管理画面で確認・解除できる |
| AUTH-005 | ログイン成功時に最終ログイン日時・IPアドレスを記録する |
| AUTH-006 | 不審なIPアドレス（初回アクセス国、短時間での連続失敗）を検知し、管理者へアラートを発する |

#### 3.1.2 多要素認証（MFA）

| ID | 要件 |
|----|------|
| MFA-001 | TOTP（RFC 6238準拠、Google Authenticator等）によるMFAを提供する |
| MFA-002 | 管理者・事務ロールはMFA必須とする（強制有効化） |
| MFA-003 | 営業・社員はMFA任意（推奨）とする |
| MFA-004 | 協力会社/個人事業主はMFA任意とする |
| MFA-005 | MFA設定時にバックアップコード（8文字×10個）を発行する。使用済みコードは無効化する |
| MFA-006 | バックアップコードを全て使用した場合、管理者によるMFAリセット手続きを経て再設定する |

#### 3.1.3 シングルサインオン（SSO）

| ID | 要件 |
|----|------|
| SSO-001 | Google Workspace（OpenID Connect / OAuth 2.0）との SSO 連携を提供する（社内ユーザー向け） |
| SSO-002 | SSO連携ユーザーはパスワード認証を無効化し、IdP認証のみとする |
| SSO-003 | SSO連携時、IdPから取得するメールアドレスをユーザーの一意識別子として使用する |
| SSO-004 | 協力会社/個人事業主はSSO対象外とし、メールアドレス＋パスワード認証のみとする |
| SSO-005 | SSOセッションとアプリケーションセッションの双方を適切にログアウトできる（SLO対応） |

#### 3.1.4 セッション管理

| ID | 要件 |
|----|------|
| SES-001 | ログイン成功時に Cognito 発行の JWT を用いて API アクセスする（トークンの保管方法はフロントエンド実装方針で決める） |
| SES-002 | アクセストークン有効期限：15分 |
| SES-003 | リフレッシュトークン有効期限：内部ユーザー8時間、外部ユーザー4時間 |
| SES-004 | 操作なし（アイドル）30分でセッションを自動失効させ、ログイン画面へリダイレクトする |
| SES-005 | ユーザーは自身の有効なセッション一覧（デバイス・最終アクセス時刻）を確認・個別失効できる |
| SES-006 | 管理者は任意のユーザーのセッションを強制失効できる |
| SES-007 | ロール変更・ユーザー無効化は次のトークンリフレッシュ時（最大15分以内）に反映される |

#### 3.1.5 パスワードポリシー

| ID | 要件 |
|----|------|
| PWD-001 | 最低12文字、大文字・小文字・数字・記号をそれぞれ1文字以上含むこと |
| PWD-002 | 過去5回分のパスワードの再利用を禁止する |
| PWD-003 | 初回ログイン時にパスワード変更を必須とする |
| PWD-004 | パスワードリセットはメールリンク方式（有効期限30分、ワンタイム）とする |
| PWD-005 | パスワードリセットメールは登録済みのメールアドレスにのみ送信する |
| PWD-006 | 定期的なパスワード変更強制は行わない（NIST SP 800-63B 準拠）。ただし漏洩検知時は強制リセットを実施する |

### 3.2 ユーザー管理

#### 3.2.1 ユーザー登録

| ID | 要件 |
|----|------|
| USR-001 | 管理者のみがユーザーを新規登録できる |
| USR-002 | 登録時に入力する情報：氏名、メールアドレス、ロール、ユーザー種別（内部/外部）、所属（社員は部署、外部は企業名） |
| USR-003 | メールアドレスはシステム全体で一意とする |
| USR-004 | 登録完了後、ユーザーへ招待メールを送信する。メール内のリンクから初回パスワードを設定させる |
| USR-005 | 招待リンクの有効期限は72時間とする。期限切れの場合は管理者が再送信できる |
| USR-006 | Team B の社員管理モジュールで社員情報が登録された際、Webhook または内部イベントで本モジュールへ通知し、自動的にユーザーアカウントを仮作成する（管理者が確認後に有効化） |
| USR-007 | 協力会社/個人事業主のユーザー登録は、Team B の協力会社管理モジュールとの連携で行う。協力会社管理で登録完了後に招待メールを送信する |

#### 3.2.2 ユーザー編集・無効化

| ID | 要件 |
|----|------|
| USR-010 | 管理者はユーザー情報（氏名・メールアドレス・ロール・所属）を編集できる |
| USR-011 | メールアドレス変更時は変更前・変更後のアドレス双方へ通知メールを送信する |
| USR-012 | ユーザーの無効化は「論理削除」とし、データは保持する。無効化ユーザーはログイン不可となり、既存セッションは即時失効する |
| USR-013 | 退職・契約終了に伴うユーザー無効化は、Team B から連携イベントを受け取り、管理者確認のうえ実施する |
| USR-014 | 無効化されたユーザーのデータは監査ログ・勤怠・請求等の参照整合性のため5年間保持する |
| USR-015 | 管理者アカウントは最低2名を維持する。最後の管理者アカウントは無効化できない |

#### 3.2.3 ユーザー検索・一覧

| ID | 要件 |
|----|------|
| USR-020 | 管理者はユーザー一覧を氏名・メールアドレス・ロール・ステータス・ユーザー種別でフィルタリング・検索できる |
| USR-021 | ユーザー一覧はCSV形式でエクスポートできる（パスワードハッシュ等の機密情報は除外） |

### 3.3 権限・ロール管理

#### 3.3.1 ロール管理

| ID | 要件 |
|----|------|
| ROLE-001 | システムには5つのシステムロール（管理者・営業・事務・社員・協力会社/個人事業主）を定義する |
| ROLE-002 | システムロールの新規追加・削除は管理者のみ可能とする（将来の拡張に備え、UI上で対応） |
| ROLE-003 | 1ユーザーに複数ロールを付与可能とする（例：社員兼営業補佐）。ただし内部ロールと外部ロールの同時付与は禁止 |
| ROLE-004 | ロールに紐づく権限の変更は、変更履歴として監査ログに記録する |

#### 3.3.2 権限設定

| ID | 要件 |
|----|------|
| PERM-001 | 権限はリソース（モジュール）×アクション（create/read/update/delete/approve）の組み合わせで定義する |
| PERM-002 | 各モジュールはAPIリクエスト時に認証サービスの権限チェックAPIを呼び出し、アクセス可否を判定する |
| PERM-003 | 「自己のみ（S）」権限は、リソースの所有者IDとリクエストユーザーIDを比較して実現する。比較ロジックは認証サービスが提供するミドルウェアで統一する |
| PERM-004 | 権限設定の変更は即時には反映されず、次のトークンリフレッシュ（最大15分）のタイミングで反映される。緊急の場合は管理者がユーザーセッションを強制失効できる |
| PERM-005 | 外部ユーザー（ROLE_PARTNER）は、内部ユーザー向けAPIエンドポイントに対してすべてアクセス禁止とする（エンドポイントレベルでの分離） |

### 3.4 セキュリティ

#### 3.4.1 監査ログ

| ID | 要件 |
|----|------|
| AUD-001 | 以下の操作は監査ログに記録する：ログイン成功/失敗、ログアウト、パスワード変更/リセット、MFA設定変更、ユーザー登録/編集/無効化、ロール変更、権限設定変更、APIトークン発行/失効、セッション強制失効 |
| AUD-002 | 監査ログに含める項目：操作日時（UTC）、操作者ユーザーID・氏名・ロール、操作種別、対象リソース・ID、操作前後の値（diff）、クライアントIPアドレス、User-Agent |
| AUD-003 | 監査ログは不変（Immutable）とし、管理者を含め削除・編集を禁止する |
| AUD-004 | 監査ログの保持期間は7年とする（電子帳簿保存法・内部統制の観点） |
| AUD-005 | 監査ログはシステム内検索（操作種別・操作者・日時範囲・対象リソース）および CSV エクスポートが可能 |

#### 3.4.2 APIトークン管理（機械間通信）

| ID | 要件 |
|----|------|
| API-001 | 各モジュール間のサーバー間通信用に、ロールと同等のスコープを持つAPIトークン（長期有効）を発行できる |
| API-002 | APIトークンはプレフィックス付き形式（例：`ck_live_xxxx`）で発行し、発行時にのみ全文を表示する。以降はマスク表示とする |
| API-003 | APIトークンには名称・説明・スコープ・有効期限（最大1年）を設定する |
| API-004 | APIトークンの利用状況（最終使用日時・使用元IP）を管理画面で確認できる |
| API-005 | APIトークンは管理者のみ発行・失効できる |
| API-006 | 90日間未使用のAPIトークンは管理者へアラートを発し、手動失効を促す |

#### 3.4.3 その他セキュリティ対策

| ID | 要件 |
|----|------|
| SEC-001 | 全通信はTLS 1.2以上を必須とする |
| SEC-002 | CSRF対策としてDouble Submit Cookie パターンを実装する |
| SEC-003 | ログインページへの不審な自動アクセスを防ぐためにreCAPTCHA v3 を導入する（スコアが閾値以下の場合はログインを拒否） |
| SEC-004 | セキュリティヘッダー（Strict-Transport-Security、Content-Security-Policy、X-Frame-Options: DENY、X-Content-Type-Options: nosniff）を全レスポンスに付与する |
| SEC-005 | 外部ユーザー向けポータルは内部ユーザー向けシステムとサブドメイン（またはパス）レベルで分離し、クロスアクセスをWebサーバー・APIゲートウェイ双方で拒否する |

## 4. 非機能要件

### 4.1 可用性

| 要件 | 内容 |
|------|------|
| 稼働率 | 99.5%以上/月（計画メンテナンス時間を除く） |
| メンテナンス | 深夜帯（22:00〜06:00 JST）に実施し、1週間前に通知 |
| 認証サービス単体障害時 | 認証サービスが停止した場合、他の全モジュールはログイン不可となるため、認証サービスは冗長構成（最低2インスタンス）を必須とする |

### 4.2 性能

| 要件 | 内容 |
|------|------|
| ログイン応答時間 | 2秒以内（MFAコード検証含む） |
| 権限チェックAPI応答時間 | 100ms以内（p99）。各モジュールから高頻度で呼び出されるため、インメモリキャッシュを活用する |
| 同時ログインユーザー数 | 最大100ユーザーを想定（将来300ユーザーまで拡張可能な設計） |

### 4.3 拡張性

| 要件 | 内容 |
|------|------|
| 外部IdP追加 | Microsoft Entra ID（旧Azure AD）等の追加SSO連携を将来的に追加可能な設計とする |
| ロール追加 | 新たなロール（例：マネージャー）を追加できる |
| SCIM対応 | ユーザープロビジョニングの自動化に向け、将来的にSCIM 2.0 APIへの対応を考慮した設計とする |

### 4.4 保守性

| 要件 | 内容 |
|------|------|
| ログ | アプリケーションログは構造化ログ（JSON形式）で出力し、ログ管理基盤（CloudWatch Logs等）に集約する |
| 設定の外部化 | セッション有効期限・ロック閾値・パスワードポリシー等はコード変更なく設定ファイルで変更可能とする |

## 5. データ項目定義

### 5.1 users（アプリケーション側ユーザメタ情報）

| カラム名 | 型 | 必須 | 説明 |
|---------|-----|------|------|
| id | UUID | Y | ユーザーID（主キー） |
| email | VARCHAR(255) | Y | メールアドレス（一意） |
| name | VARCHAR(100) | Y | 氏名 |
| cognito_sub | VARCHAR(64) | Y | Cognito の subject（ユーザー一意識別子） |

## 7. 他モジュールとの連携インターフェース

### 7.1 本モジュールが提供するAPI（他チームが利用）

#### 7.1.1 権限チェックAPI

```
POST /internal/auth/check-permission
Authorization: Bearer <APIトークン>

Request:
{
  "user_id": "uuid",
  "resource": "timesheet",
  "action": "read",
  "resource_owner_id": "uuid"  // scope=self の場合に使用
}

Response:
{
  "allowed": true | false,
  "reason": "string"  // 不許可の場合の理由（ログ用）
}
```

- **呼び出し元**: Team B（社員管理）、Team C（案件管理）、Team D（勤怠・請求）、Team E（書類・通知）の全API
- **性能要件**: 応答100ms以内（p99）。クライアント側でのキャッシュ（TTL 60秒）を推奨

#### 7.1.2 ユーザー情報取得API

```
GET /internal/auth/users/{user_id}
Authorization: Bearer <APIトークン>

Response:
{
  "id": "uuid",
  "name": "string",
  "email": "string",
  "roles": ["ROLE_EMPLOYEE"],
  "user_type": "internal",
  "status": "active"
}
```

- **呼び出し元**: Team B〜E の全モジュール（表示名取得・ユーザー種別判定等）

#### 7.1.3 ユーザー存在確認・一覧取得API

```
GET /internal/auth/users?ids=uuid1,uuid2,...
Authorization: Bearer <APIトークン>
```

- **用途**: Team D（勤務表承認フロー等でユーザー情報を一括取得）、Team E（通知送信先のユーザー情報取得）

#### 7.1.4 トークン検証ミドルウェア（共有ライブラリ）

- 各モジュールが組み込むJWT検証ミドルウェアをライブラリとして提供する
- 検証内容：署名の正当性、有効期限、ユーザーステータス（無効化チェック）
- 言語：TypeScript/Node.js 向けを優先提供。他言語は要調整

### 7.2 本モジュールが受け取るイベント（他チームから連携）

| イベント種別 | 連携元 | 内容 | 処理 |
|------------|--------|------|------|
| employee.created | Team B（社員管理） | 社員情報登録完了 | ユーザーアカウントを仮作成（status: invited） |
| employee.offboarded | Team B（社員管理） | 退職・契約終了 | 管理者へ無効化タスクを通知 |
| partner.registered | Team B（協力会社管理） | 協力会社/個人事業主登録完了 | 外部ユーザーアカウント仮作成・招待メール送信 |
| partner.contract_ended | Team B（協力会社管理） | 契約終了 | 管理者へ無効化タスクを通知 |

- イベント連携方式：内部メッセージキュー（例：AWS SQS）または Webhook（要Team Bと合意）

### 7.3 Team E（通知・ワークフロー）との連携

- 認証関連の通知（パスワードリセット、アカウントロック等）は、Team E の通知基盤を利用せず本モジュールが直接メール送信する（認証基盤の独立性・信頼性確保のため）
- セキュリティアラート（不審ログイン、管理者MFA解除依頼等）は Team E の通知基盤を経由して管理者へ通知する

## 8. ユースケース（主要シナリオ）

### UC-001: 社員の初回ログイン

```
前提条件: 管理者が社員ユーザーを登録済み（またはTeam B連携で仮作成済み）

1. 社員はメールで受け取った招待リンクをクリックする
2. 初回パスワード設定画面で新しいパスワードを設定する
3. ログイン画面でメールアドレスとパスワードを入力する
4. ログイン成功。ダッシュボードへ遷移する
5. MFA設定を推奨するバナーが表示される

事後条件: 社員がシステムにアクセスできる状態になる
```

### UC-002: 管理者によるロール付与

```
前提条件: 管理者がログイン済み

1. ユーザー一覧画面から対象ユーザーを検索・選択する
2. ユーザー詳細画面でロール編集ボタンをクリックする
3. 付与するロールを選択し、保存する
4. 権限変更が監査ログに記録される
5. 対象ユーザーへ通知メールが送信される
6. 対象ユーザーのトークンリフレッシュ時（最大15分以内）に新ロールが反映される

事後条件: 対象ユーザーが新しいロールでシステムを利用できる
```

### UC-003: 協力会社担当者のログインと請求書提出

```
前提条件: Team B で協力会社登録済み。外部ユーザーとして招待メール送信済み

1. 協力会社担当者は外部ポータルURL（例：partner.company-k.example.com）にアクセスする
2. メールアドレスとパスワードでログインする
3. ダッシュボードには自身が参画中の案件情報・勤務表・請求書のみ表示される
4. 当月勤務表を入力し、提出する（Team D 勤怠管理モジュール）
5. 請求書ファイルをアップロードする（Team D 請求管理モジュール）
6. 内部業務画面（社員一覧、他社案件等）へのアクセスを試みても403エラーとなる

事後条件: 勤務表と請求書が提出される。内部情報への不正アクセスはブロックされる
```

### UC-004: ユーザーアカウントのロックと解除

```
前提条件: 該当ユーザーが存在する

1. ユーザーがパスワードを5回連続で間違える
2. アカウントが30分間ロックされる
3. ロック中はログイン不可となり「アカウントがロックされています」メッセージを表示
4a. [自動解除] 30分後、ロックが自動解除され通常ログイン可能になる
4b. [手動解除] 管理者がユーザー詳細画面からロック解除ボタンをクリックする

事後条件: ユーザーがログイン可能な状態に戻る
```

### UC-005: 退職社員のアカウント無効化

```
前提条件: 社員が退職手続き完了。Team B の社員管理で退職処理済み

1. Team B から employee.offboarded イベントが発行される
2. 本モジュールが受信し、管理者へ「アカウント無効化タスク」通知を送る
3. 管理者はユーザー詳細画面でアカウントを無効化する
4. 既存の全セッションが即時失効する
5. 以降、該当ユーザーはログイン不可となる
6. 無効化操作が監査ログに記録される

事後条件: 退職社員がシステムにアクセスできない状態になる。データは保持される
```

### UC-006: APIトークン発行（モジュール間連携用）

```
前提条件: 管理者がログイン済み。新しいモジュールが本番環境にデプロイされる

1. 管理者はAPIトークン管理画面を開く
2. トークン名・説明・スコープ・有効期限を設定して発行する
3. 発行直後のみトークン全文が表示される。管理者はトークンをコピーして安全に保管する
4. 新モジュールの設定ファイルにトークンを設定する
5. 以降、新モジュールは権限チェックAPIをトークンで呼び出せる

事後条件: 新モジュールが認証サービスのAPIを利用できる
```

## 9. 制約・前提条件

### 9.1 技術的制約

| 制約 | 内容 |
|------|------|
