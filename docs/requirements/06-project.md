# 案件管理 - 要件定義

## 1. モジュール概要

### 1.1 目的
SES（客先常駐）および受託開発の案件情報を一元管理し、商談から受注・完了までのライフサイクル全体を追跡する。また、顧客請求単価と社員/協力会社支払単価の差分管理を行い、収益性の把握を支援する。

### 1.2 対象ユーザー
| ロール | 操作権限 |
|--------|----------|
| 管理者 | 全機能（参照・作成・編集・削除・単価情報閲覧） |
| 営業 | 参照・作成・編集（削除は不可、単価情報閲覧可） |
| 事務 | 参照・編集（契約・請求引き継ぎ処理、単価情報閲覧可） |
| 社員 | 自分が参画中の案件のみ参照（単価情報は非表示） |
| 協力会社/個人事業主 | 公開案件の閲覧のみ（個人情報・単価は非表示） |

### 1.3 案件種別
- **SES案件:** 客先への技術者常駐。月単位の継続契約が基本。稼働工数に応じた請求。
- **受託開発案件:** 成果物（システム・アプリ等）の納品を目的とする請負契約。マイルストーン・フェーズ管理が必要。

---

## 2. 案件ライフサイクル

### 2.1 ステータス遷移図

```
[商談中] → [提案中] → [受注] → [進行中] → [完了]
    ↓           ↓          ↓
  [失注]      [失注]     [中止]
```

### 2.2 各ステータスの定義

| ステータス | 説明 | 主な担当者 |
|------------|------|------------|
| 商談中 | 顧客との初期商談フェーズ。要件ヒアリング中 | 営業 |
| 提案中 | 提案書・見積書を提出し、顧客の判断待ち | 営業 |
| 受注 | 顧客から正式受注。契約手続き開始 | 営業 → 事務 |
| 進行中 | 技術者アサインまたは開発作業が開始済み | 営業・事務・PM |
| 完了 | 納品完了または契約終了 | 事務 |
| 失注 | 顧客が競合他社を選択または案件自体がなくなった | 営業 |
| 中止 | 受注後に案件がキャンセルになった | 営業・事務 |

### 2.3 ステータス遷移ルール
- 「受注」→「進行中」への遷移時は、Team D（勤怠・請求）への契約情報引き継ぎが必要
- 「失注」「中止」時は失注理由の記録を必須とする
- ステータス変更は変更履歴として記録する（誰が・いつ・どのステータスから変更したか）

## 3. 機能要件

### 3.1 案件CRUD

#### 3.1.1 案件登録
- 案件種別（SES / 受託開発）の選択
- 案件名・概要・顧客企業・担当営業の登録
- 案件種別に応じた追加項目の入力（後述のデータ項目定義参照）
- 顧客請求単価と支払単価の登録（複数人・フェーズに対応）
- 添付ファイル（提案書・見積書・契約書等）の管理

#### 3.1.2 案件編集
- 登録情報の変更（変更履歴を記録）
- ステータス変更（遷移ルールに従う）
- アサインメンバーの追加・変更・削除

#### 3.1.3 案件削除・アーカイブ
- 論理削除（アーカイブ）のみ。物理削除は管理者のみ
- 「進行中」ステータスの案件は削除不可

### 3.2 ステータス管理

#### 3.2.1 ステータス変更
- ステータス変更フォームから変更理由・次のアクションを記録
- 「受注」→「進行中」への変更時：Team D への引き継ぎフラグを自動設定
- 「失注」「中止」への変更時：失注理由（自由記述＋選択肢）の入力を必須化

#### 3.2.2 変更履歴管理
- ステータス変更・主要項目変更の履歴を自動記録
- 変更者・変更日時・変更前後の値を保持

### 3.3 案件DB検索

#### 3.3.1 基本検索
- 案件名（部分一致）
- 顧客企業名
- 案件種別（SES / 受託開発）
- ステータス（複数選択可）
- 担当営業

#### 3.3.2 詳細絞り込み
- 案件開始予定日・終了予定日の範囲指定
- スキルタグによる絞り込み（SES案件：必要スキルで検索）
- 単価レンジ（顧客請求単価）での絞り込み
- アサイン済み / 未アサインのフィルター

#### 3.3.3 案件一覧表示
- 一覧でのソート（日付・ステータス・案件種別）
- カード形式 / テーブル形式の切り替え
- CSV エクスポート（権限に応じて単価情報の出力有無を制御）

### 3.4 単価差分管理（収益管理）

- 顧客請求単価（クライアントへの請求額）
- 支払単価（社員 / 協力会社への支払額）
- 差分（粗利）の自動計算・表示
- 閲覧権限は管理者・営業・事務に限定

## 7. ユースケース

### UC-PRJ-01: SES案件の新規登録
**アクター:** 営業  
**事前条件:** 対象顧客企業が登録済み、営業ロールでログイン済み  
**主フロー:**
1. 案件一覧から「新規登録」を選択
2. 案件種別「SES」を選択
3. 案件名・顧客企業・就業場所・必要スキル・必要人数・精算条件を入力
4. 顧客請求単価（月単価）を設定
5. ステータスを「商談中」で保存
6. 必要に応じて提案書をファイル添付

---

### UC-PRJ-02: 受託開発案件の受注処理
**アクター:** 営業  
**事前条件:** 案件ステータスが「提案中」  
**主フロー:**
1. 案件詳細画面でステータスを「受注」に変更
2. 契約金額・支払条件・成果物定義・検収条件を入力
3. 支払単価（開発メンバー別）を登録
4. 保存後、Team D（勤怠・請求）への引き継ぎ通知が自動送信される

---

### UC-PRJ-03: 失注の記録
**アクター:** 営業  
**事前条件:** 案件ステータスが「商談中」または「提案中」  
**主フロー:**
1. 案件詳細でステータス変更フォームを開く
2. 「失注」を選択
3. 失注理由（選択肢から選択 + 自由記述）を入力
4. 保存。案件は失注ステータスで保持（削除はしない）

**失注理由選択肢例:**
- 価格競争負け
- 競合他社選定
- 案件自体の中止
- 顧客都合
- その他

---

### UC-PRJ-04: アサインメンバーの追加
**アクター:** 営業  
**事前条件:** 案件ステータスが「受注」または「進行中」、マッチングでメンバー候補確定済み  
**主フロー:**
1. 案件詳細のアサイン管理画面を開く
2. マッチングモジュールまたは直接指定でアサイン候補を選択
3. 参画開始日・終了予定日・役割を設定
4. 支払単価を設定（project_ratesに保存）
5. ステータスを「confirmed（確定）」で保存

## 8. 制約・前提条件

### 8.1 案件種別ごとの制約
- SES案件：月単価が基本。精算幅（下限・上限工数）を設定する場合はその旨を明記
- 受託開発案件：契約金額を必須とし、マイルストーン単位の部分支払いにも対応

### 8.2 単価情報の権限制約
- 顧客請求単価・支払単価・差分（粗利）は管理者・営業・事務のみ閲覧可能
- 社員・協力会社は自分の支払単価も閲覧不可（契約書は別途書類管理モジュールで管理）

### 8.3 データ保持ポリシー
- 完了・失注・中止の案件は論理削除のみ。受注履歴・請求履歴との整合性を保つため物理削除不可
- ステータス変更履歴は永続保持

### 8.4 性能要件
- 案件一覧（全ステータス）の初期表示：2秒以内（5,000件以上を想定）
- 案件検索・絞り込み：1秒以内

### 8.5 前提条件
- Team A（認証・権限）のロール管理が完成していること
- Team B（社員・スキル）のスキルタグ定義が確定していること（スキルタグの体系を共通化する必要あり）
- Team D（勤怠・請求）との引き継ぎWebhookの仕様は両チームで別途合意すること
