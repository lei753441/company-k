# ダッシュボード・レポート - 要件定義

## 1. モジュール概要

### 1.1 目的
IT開発会社（SES・受託開発）の経営KPI・稼働状況・売上・案件進捗を可視化し、管理者・営業・事務の意思決定を支援する。各業務モジュールのデータを集約・集計してロール別ダッシュボードおよびレポートとして提供する。

### 1.2 設計方針
- **ロール別表示**: 各ロールに必要な情報のみを表示し、権限外データは非表示とする
- **データ鮮度の明示**: 表示データの最終集計日時を常にダッシュボード上に明示する
- **ドリルダウン**: サマリー数値から詳細画面へのリンクを提供する
- **レポート出力**: CSV・PDF形式でのデータエクスポートを提供する

### 1.3 対象ユーザー

| ロール | 主な利用内容 |
|--------|------------|
| 管理者 | 全社稼働率・売上・コスト・案件サマリーの把握、経営判断 |
| 営業 | 案件パイプライン・提案状況・成約率の把握 |
| 事務 | 月次請求・支払サマリー・勤怠承認状況の確認 |
| 社員 | スコープ外（個人画面は各業務モジュール側で提供） |
| 協力会社 | スコープ外 |

## 3. レポート一覧

### 3.1 稼働率レポート

- **目的**: 社員・協力会社の稼働状況を時系列で把握する
- **対象ロール**: 管理者・営業
- **集計期間**: 月次・四半期・年次（選択可）
- **フィルター**: 対象期間・部署・雇用形態（正社員/協力会社）
- **表示内容**:
  - 期間別稼働率推移グラフ
  - 個人別稼働状況一覧（社員名・稼働期間・プロジェクト名・稼働率）
  - 空き期間（ベンチ期間）の集計
- **出力形式**: CSV・PDF

### 3.2 売上・利益レポート

- **目的**: 案件・顧客・月別の売上・コスト・粗利を把握する
- **対象ロール**: 管理者
- **集計期間**: 月次・四半期・年次（選択可）
- **フィルター**: 対象期間・顧客・案件・担当営業
- **表示内容**:
  - 月次売上・コスト・粗利の推移グラフ
  - 顧客別売上ランキング
  - 案件別売上・粗利一覧
  - 担当営業別売上サマリー
- **出力形式**: CSV・PDF

### 3.3 案件進捗レポート

- **目的**: 案件のパイプラインと進捗状況を把握する
- **対象ロール**: 管理者・営業
- **集計期間**: 当月・当四半期・カスタム期間
- **フィルター**: 案件ステータス・顧客・担当営業・期間
- **表示内容**:
  - ステータス別案件件数・金額
  - 案件一覧（案件名・顧客・担当・ステータス・期間・金額）
  - 失注案件の理由別集計
- **出力形式**: CSV・PDF

### 3.4 カスタムレポート（CSV/PDF出力）

- **目的**: ユーザーが必要な項目を選択して自由にレポートを生成する
- **対象ロール**: 管理者・事務
- **機能**:
  - データソースを選択する（社員・案件・請求・勤怠）
  - 表示列を選択する
  - フィルター条件を設定する
  - 並び替え条件を設定する
  - レポートテンプレートとして保存・再利用する
- **出力形式**: CSV・PDF
- **制約**: 個人情報を含む列は管理者のみ選択可能

## 4. KPI定義（各指標の計算方法）

### 4.1 稼働率

```
全社稼働率（%）= 稼働中人数 ÷ 全アクティブ社員数 × 100

- 稼働中人数: 当日時点で案件に紐づきステータスが「進行中」の社員・協力会社数
- 全アクティブ社員数: 退職・契約終了していない全社員・協力会社数
- 小数点第1位まで表示

例: 稼働中20名 ÷ アクティブ25名 × 100 = 80.0%
```

```
個人稼働率（%）= 当月稼働時間 ÷ 当月所定労働時間 × 100

- 当月稼働時間: 勤務表の承認済み実稼働時間
- 当月所定労働時間: 就業規則に基づく月の標準労働時間（例: 160時間）
- 小数点第1位まで表示
```

### 4.2 売上・利益指標

```
当月売上（円）= Σ（承認済み請求書の請求金額） ※当月発行分

当月コスト（円）= Σ（当月支払い予定の協力会社支払金額）
              + Σ（当月確定した社員給与・外注費）

当月粗利（円）= 当月売上 - 当月コスト

粗利率（%）= 当月粗利 ÷ 当月売上 × 100
```

```
顧客別売上（円）= Σ（対象顧客向け承認済み請求書の請求金額）

案件別粗利（円）= 案件の請求金額合計 - 案件に紐づく外注費・人件費合計
```

### 4.3 案件パイプライン指標

```
成約率（%）= 当月受注件数 ÷ 当月クローズ件数（受注＋失注）× 100

- 当月受注件数: ステータスが「受注」に変化した案件数
- 当月クローズ件数: ステータスが「受注」または「失注」に変化した案件数
```

```
パイプライン総額（円）= Σ（商談中・提案中案件の予定金額）

加重パイプライン（円）= Σ（案件予定金額 × 成約確度[%]）
- 成約確度は案件登録時に営業が設定する（10%〜100%の10%刻み）
```

### 4.4 請求・支払指標

```
当月請求合計（円）= Σ（当月発行済み請求書金額）

未収金額（円）= Σ（入金期限を過ぎた未入金請求書金額）

当月支払合計（円）= Σ（当月支払い予定の協力会社請求書金額）
```

### 4.5 勤務表承認指標

```
勤務表提出率（%）= 当月提出済み勤務表数 ÷ 当月提出対象者数 × 100

勤務表承認率（%）= 当月承認済み勤務表数 ÷ 当月提出済み勤務表数 × 100

- 提出対象者: 当月稼働中の全社員・協力会社
```

## 5. データ項目定義

| dimension_key | VARCHAR(50) | - | ディメンションキー（例: customer_id） |
| dimension_value | VARCHAR(100) | - | ディメンション値（例: 顧客UUID） |
| created_at | TIMESTAMP | ○ | 作成日時 |

| 項目名 | 型 | 必須 | 説明 |

## 7. 他モジュールとのデータ取得インターフェース

ダッシュボード・レポートモジュールは各業務モジュールから定期的にデータを取得・集計する。リアルタイムデータは各モジュールのAPIを直接参照し、日次集計データはバッチでスナップショットとして保存する。

### 7.1 Team B（社員・スキル）から取得するデータ

| データ | 取得方法 | 取得タイミング | 用途 |
|--------|---------|------------|------|
| 社員一覧（ID・名前・雇用形態・在籍状況） | REST API | 日次バッチ | 稼働率計算の分母 |
| 稼働状況（稼働中/空き/稼働終了予定日） | REST API | 日次バッチ | 稼働率・空き要員数 |

- `GET /api/employees?status=active` → アクティブ社員一覧
- `GET /api/employees/occupancy-summary` → 稼働状況サマリー

### 7.2 Team C（案件管理）から取得するデータ

| データ | 取得方法 | 取得タイミング | 用途 |
|--------|---------|------------|------|
| 案件一覧（ID・名前・ステータス・金額・期間・顧客） | REST API | 日次バッチ + リアルタイム | パイプライン・案件KPI |
| 案件ステータス変更履歴 | REST API / イベント | 日次バッチ | 成約率計算 |

- `GET /api/projects?updated_after={date}` → 更新された案件一覧
- `GET /api/projects/pipeline-summary` → パイプラインサマリー

### 7.3 Team D（勤怠管理）から取得するデータ

| データ | 取得方法 | 取得タイミング | 用途 |
|--------|---------|------------|------|
| 月次勤務表サマリー（提出数・承認数・未提出数） | REST API | リアルタイム | 勤務表承認状況KPI |
| 個人別稼働時間 | REST API | 月次バッチ | 稼働率レポート |

- `GET /api/attendance/monthly-summary?year={YYYY}&month={MM}` → 月次勤務表サマリー

### 7.4 Team D（請求・支払管理）から取得するデータ

| データ | 取得方法 | 取得タイミング | 用途 |
|--------|---------|------------|------|
| 月次請求合計・支払合計 | REST API | 日次バッチ | 請求・支払KPI |
| 請求書一覧（金額・ステータス・入金状況） | REST API | 日次バッチ | 未収金・売上集計 |
| 協力会社支払一覧（金額・支払予定日） | REST API | 日次バッチ | コスト集計 |

- `GET /api/invoices?month={YYYY-MM}` → 月次請求書一覧
- `GET /api/payments?month={YYYY-MM}` → 月次支払一覧

### 7.5 Team D（契約管理）から取得するデータ

| データ | 取得方法 | 取得タイミング | 用途 |
|--------|---------|------------|------|
| 期限到来契約一覧（契約名・期限日） | REST API | 日次バッチ | 事務ダッシュボード・アラート |

- `GET /api/contracts?expires_within_days=30` → 30日以内に期限を迎える契約

## 8. ユースケース

### UC-DSH-001: 管理者が月初に全社稼働率を確認する

1. 管理者がシステムにログインし、管理者ダッシュボード（SCR-DSH-001）が表示される
2. WGT-ADM-001で全社稼働率（例：80.0%）と前月比（例：+2.5%）を確認する
3. 数値をクリックすると稼働率レポート（SCR-DSH-004）にドリルダウンし、個人別の稼働状況を確認する
4. 空き要員数（WGT-ADM-007）を確認し、営業に対して案件マッチングを促す

### UC-DSH-002: 営業が当月のパイプライン状況を確認する

1. 営業担当がシステムにログインし、営業ダッシュボード（SCR-DSH-002）が表示される
2. WGT-SLS-001で商談中・提案中案件の件数と合計金額を確認する
3. WGT-SLS-004のパイプライングラフで各ステージの案件分布を確認する
4. WGT-SLS-003で当月成約率を確認し、進捗が遅い場合は重点案件を特定する

### UC-DSH-003: 事務担当が月末に勤務表承認状況を確認する

1. 事務担当がシステムにログインし、事務ダッシュボード（SCR-DSH-003）が表示される
2. WGT-ADM_JIMU-004で当月の勤務表提出済み件数・承認済み件数・未提出件数を確認する
3. WGT-ADM_JIMU-005の未承認勤務表一覧から、対応が必要な勤務表を確認する
4. 一覧から直接承認画面（通知・ワークフローモジュール）に遷移して承認操作を行う

### UC-DSH-004: 管理者が月次売上・利益レポートをPDF出力する

1. 管理者が売上・利益レポート画面（SCR-DSH-005）を開く
2. 対象期間（例：2024年1月〜12月）・顧客（全顧客）を選択する
3. 「PDF出力」ボタンを押す
4. バックグラウンドでPDF生成処理が開始され、完了後にダウンロードリンクが通知される
5. 管理者がPDFをダウンロードし、経営報告に使用する

### UC-DSH-005: 事務担当がカスタムレポートを作成して再利用する

## 9. 制約・前提条件

### 9.1 データ更新頻度・集計タイミング

| データ種別 | 更新方式 | タイミング | 説明 |
|-----------|---------|---------|------|
| リアルタイムKPI（案件数・タスク数） | API直接参照 | 画面表示時 | キャッシュ有効期限5分 |
| 日次集計KPI（稼働率・売上等） | バッチ集計 | 毎朝6:00 | 前日データをスナップショット保存 |
| 月次集計データ | バッチ集計 | 毎月1日 7:00 | 前月確定データを月次テーブルに保存 |
| レポート出力データ | オンデマンド生成 | ユーザー操作時 | 大量データは非同期処理（5分以内完了目安） |

### 9.2 表示データの最終更新日時明示
- 各ウィジェット・レポートには「データ最終更新：YYYY-MM-DD HH:MM」を必ず表示する
- バッチ失敗時はウィジェットに「データ更新中」バナーを表示し、前回の値を維持する

### 9.3 パフォーマンス
- ダッシュボード初回表示：3秒以内（全ウィジェット読み込み完了まで）
- レポート画面表示：5秒以内（1,000件以下のデータ）
- CSV出力（10,000件以下）：1分以内
- PDF出力：3分以内（非同期処理でバックグラウンド実行）

### 9.4 権限・セキュリティ
- 管理者ダッシュボードは管理者ロールのみアクセス可能
- 営業ダッシュボードは営業ロールのみアクセス可能（管理者も閲覧可能）
- 事務ダッシュボードは事務ロールのみアクセス可能（管理者も閲覧可能）
- カスタムレポートで個人情報を含む列（社員の住所・給与等）は管理者のみ選択可能
- レポート出力ファイルは生成後24時間で自動削除する（再出力が必要）

### 9.5 前提条件
- 各業務モジュール（Team B・C・D）が定義のAPIエンドポイントを提供していることを前提とする
- 日次・月次バッチはシステムスケジューラー（cron等）で管理し、バッチ失敗時はアラートを管理者に通知する
- グラフ描画はフロントエンドのチャートライブラリ（例：Chart.js、Recharts等）を使用する
- PDF出力はサーバーサイドレンダリング（例：WeasyPrint、puppeteer等）を使用する
- CSV出力はUTF-8（BOM付き）で出力してExcelでの文字化けを防ぐ
